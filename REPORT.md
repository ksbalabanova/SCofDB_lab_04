## 1. Постановка сценария

Нужно отразить:
1. Что происходит без защиты.
Клиент отправляет запрос на оплату заказа. Сервер успешно обрабатывает его и фиксирует оплату в БД. В момент отправки ответа происходит обрыв сети — клиент получает таймаут и не знает, прошла ли оплата. Клиент повторяет запрос, но без защиты сервер обрабатывает его как новый.
2. Почему возможна повторная обработка одного и того же намерения клиента.
Клиент не может отличить «запрос не дошёл» от «ответ потерялся». Без идемпотентности повторный запрос приводит к ошибке, и клиент ничего не знает

## 2. Реализация таблицы idempotency_keys

```sql
CREATE TABLE idempotency_keys (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    idempotency_key VARCHAR(255) NOT NULL,
    request_method VARCHAR(16) NOT NULL,
    request_path TEXT NOT NULL,
    request_hash TEXT NOT NULL,
    status VARCHAR(32) NOT NULL DEFAULT 'processing',
    status_code INTEGER,
    response_body JSONB,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    expires_at TIMESTAMP NOT NULL,
    CONSTRAINT idempotency_status_check CHECK (status IN ('processing', 'completed', 'failed')),
    CONSTRAINT idempotency_keys_unique UNIQUE (idempotency_key, request_method, request_path)
);

CREATE INDEX idx_idempotency_lookup
    ON idempotency_keys (idempotency_key, request_method, request_path);

CREATE INDEX idx_idempotency_expires
    ON idempotency_keys (expires_at);
```

Поля idempotency_key, request_method и request_path вместе идентифицируют запрос — уникальный ключ строится на тройке, а не только на ключе клиента. Поле request_hash хранит SHA256 от тела запроса и позволяет обнаружить попытку переиспользовать ключ с другим payload. Поля status_code и response_body хранят кэш ответа для возврата при повторе. Поле expires_at ограничивает время жизни записи 24 часами.

Уникальный constraint защищает от дублирования на уровне БД. Индекс idx_idempotency_lookup ускоряет поиск при каждом входящем запросе, индекс idx_idempotency_expires — периодическую очистку устаревших записей.

## 3. Реализация middleware

1. Чтение Idempotency-Key. 
Middleware проверяет метод запроса и путь — обрабатываются только POST на платёжные эндпоинты. Если заголовок Idempotency-Key отсутствует, запрос передаётся дальше без изменений.

2. Проверка существующей записи. 
Читается тело запроса, вычисляется SHA256-хэш. В БД выполняется поиск записи по тройке (ключ, метод, путь). Если запись найдена со статусом completed — проверяется совпадение хэша.

3. Обработка кейса "тот же key + другой payload". 
Если запись найдена, но хэш тела отличается от сохранённого — возвращается 409 Conflict с сообщением о конфликте. Бизнес-логика не вызывается.

4. Сохранение результата первого запроса. 
Если запись не найдена — вставляется новая со статусом processing. Выполняется реальный запрос через call_next. После получения ответа тело читается, запись в БД обновляется до статуса completed с сохранением status_code и response_body.

5. Возврат кэшированного ответа при повторе. 
Если запись найдена со статусом completed и хэш совпадает — бизнес-логика не вызывается, клиенту возвращается кэшированный ответ с заголовком X-Idempotency-Replayed: true.

## 4. Демонстрация без защиты

Имитируется сценарий сетевого сбоя: клиент отправляет запрос на оплату, не получает ответ и повторяет его без Idempotency-Key. Первый запрос успешно оплачивает заказ. Второй запрос получает бизнес-ошибку — заказ уже оплачен. Оба запроса вернули HTTP 200, но с разным полем success. Middleware не вмешивался — заголовок X-Idempotency-Replayed отсутствует. Клиент не может отличить успешную оплату от ошибки повтора.

## 5. Демонстрация с Idempotency-Key

Имитируется тот же сценарий сетевого сбоя, но клиент передаёт одинаковый Idempotency-Key в обоих запросах. Первый запрос обработан сервисом, результат сохранён в таблице idempotency_keys со статусом completed. Второй запрос перехвачен middleware — бизнес-логика не вызывалась, вернулся кэшированный ответ с заголовком X-Idempotency-Replayed: true. В таблице order_status_history зафиксирована одна запись об оплате — повторного списания не произошло.

## 7. Сравнение с решением из ЛР2 (FOR UPDATE)

<img width="622" height="539" alt="Снимок экрана 2026-04-03 в 23 05 24" src="https://github.com/user-attachments/assets/3b7047ed-49a9-4a8d-9713-83e5d369066e" />


FOR UPDATE — это блокировка строки в БД на уровне транзакции. Она защищает от гонки при одновременных запросах: второй запрос ждёт завершения первого, после чего видит уже оплаченный заказ и получает бизнес-ошибку. Гарантия обеспечивается на уровне БД.

Idempotency Key — это кэширование результата на уровне API. Он защищает от повторных запросов после сетевых сбоев: клиент получает тот же успешный ответ что и в первый раз, не зная что это был кэш. Гарантия обеспечивается на уровне middleware.

При повторе запроса FOR UPDATE возвращает бизнес-ошибку — клиент вынужден самостоятельно понимать что оплата всё-таки прошла. Idempotency Key возвращает прозрачный успех — клиент не замечает что произошёл retry.

Механизмы решают разные проблемы и не заменяют друг друга. FOR UPDATE нужен для защиты от одновременных параллельных запросов, Idempotency Key — для защиты от последовательных повторов после сбоев. В продакшн-системе следует использовать оба.

## 8. Выводы

1. Таймаут на стороне клиента не означает что операция не выполнилась — сервер мог обработать запрос и потерять только ответ.
2. Idempotency Key позволяет клиенту безопасно повторять запросы не опасаясь двойного списания.
3. Кэш ответа хранится в БД с TTL 24 часа — этого достаточно для retry-сценариев, но не создаёт бесконечного роста таблицы.
4. FOR UPDATE и Idempotency Key решают разные проблемы: первый защищает от параллельных запросов, второй — от последовательных повторов.
5. Проверка хэша payload при повторном использовании ключа предотвращает молчаливые ошибки когда клиент случайно отправляет другие данные с тем же ключом.
